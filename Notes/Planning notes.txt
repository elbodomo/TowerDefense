Project Development Notes: 10-Hour Sprint
1. Development Sessions
Thursday (~30 min): Initial planning, Unity version setup (6000.0.58f2), and core architecture ideation.

Friday (2:30h): Core project structure, Singleton managers, and basic game flow setup.

Saturday (3:00h): Tower logic, tower selection UI, and UIManager implementation.

Sunday (2:00h): JSON-based Wave Manager, enemy prefab creation, and Start/Stop game state logic.

Monday (2:15h): Gameplay balancing, low-poly 3D modeling, particle feedback systems, level design, and WebGL build.

2. Architecture & Performance
Enemy System: Built on an abstract EnemyBase to handle health, movement, and death events. Includes "Scout" (speed-focused) and "Heavy" (health-focused) variants.

Tower System: Abstract TowerBase manages enemy detection via LayerMasks and fire-rate cooldowns.

Tower Types: Features a Sniper (high-precision single-target), AOE (radius-based splash damage), and Support (passive economy generation).

Optimization: Implemented Object Pooling for all enemy types to ensure stable and smooth performance for the WebGL target.

3. Implementation Details
Visual Feedback: While visual experience wasn't strictly required, I implemented muzzle flashes, impact particles, and material color swaps (Damage Flashes) to provide essential gameplay feedback.

Level and Wave Loading: Utilized JSON files to handle level layouts and wave data. This allowed for rapid iteration and testing by leveraging LLM efficiency with structured data.

Navigation: Created a waypoint-based system where enemies calculate the path to the base and rotate to face their direction of travel.

4. Challenges & Meandering
Pivoting the wave Builder: I initially attempted to build a manual wave-editing tool for use in the engine, but realized this was consuming too much of the 10-hour budget. I pivoted to the JSON-loading system, which proved much faster to iterate on and easier to scale.

Object Pooling Scalability: My current pooling implementation uses separate lists for each enemy type. A more scalable approach for a larger game would be a generic Dictionary-based pool to reduce boilerplate code when adding new unit types.
